What happens behind the scenes (invisible to officer)

1. Backend generates signed PDF certificate → stores in cloud storage

2. Backend calls Blockchain Integration Service:

   POST /api/chain/mint

   { certNumber: "TVET-2026-123456",

     program: "VEI", trade: "Electrical Installation",

     centerId: "LG-VEI-001", issueDate: "2026-04-13",

     approvalRef: "NAB-APR-78291",

     pdfHash: "sha256:a8f3c9..." }

3. Service mints compressed NFT on Solana via Bubblegum

4. cNFT asset ID stored in PostgreSQL alongside certificate record

5. Total time: ~2 seconds. Officer never waits.

The officer's workflow is identical whether blockchain is enabled or not. The on-chain mint happens asynchronously — if it fails, it retries silently. The certificate is issued either way, and the chain record catches up.

What happens behind the scenes

1. Frontend sends: POST /api/certificate/verify

   { certNumber: "TVET-2026-123456" }

2. Backend → Blockchain Integration Service

3. Service queries Helius DAS API for cNFT matching certNumber

4. Retrieves on-chain metadata + Merkle proof

5. Cross-references pdfHash on-chain vs. hash in PostgreSQL

6. Both match → returns "VALID"

7. Holder name pulled from PostgreSQL (never on-chain)

8. Total time: ~1.5 seconds

The employer never knows blockchain is involved. They see a government verification portal, enter a number, and get a result. The on-chain lookup is what guarantees the result hasn't been fabricated — but that's infrastructure, not interface.

How "chain-verified" works

1. Backend periodically checks each cert's on-chain status

2. Queries Helius: does cNFT exist with matching certNumber?

3. Compares on-chain pdfHash with stored pdfHash

4. If both match → "Chain-verified" (green)

5. If mismatch → "Integrity alert" (red) — triggers investigation

6. If not yet anchored → "Pending" (amber) — retry in progress

The FME admin dashboard is the only interface that acknowledges the blockchain layer exists — and even there, it's presented as a system health metric and a data integrity indicator, not as a separate module the admin has to manage.
The "blockchain module" doesn't look like a module at all.
It looks like a small badge, a system health indicator, and a verification result that happens to be powered by an on-chain lookup instead of a database query. Everything else is plumbing.

The design principle
Blockchain is infrastructure, not interface. Like PostgreSQL or a CDN, it operates silently behind the API gateway. Nobody asks users to interact with their database directly — they shouldn't interact with the blockchain directly either. The entire Solana layer sits behind the Blockchain Integration Service, and no user — across any of the five portals — ever touches a wallet, signs a transaction, pays gas, or sees a blockchain explorer. The UX is 100% web2. The trust guarantees are web3.
